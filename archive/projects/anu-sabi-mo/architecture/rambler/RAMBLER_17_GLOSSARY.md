# Rambler Glossary

> *This glossary defines the core architectural vocabulary used throughout the Rambler Engine documentation. It provides a common language for contributors and helps ensure consistent interpretation of architectural concepts.*

---

# Purpose

Architecture is easiest to understand when important terms have precise meanings.

This glossary establishes a shared vocabulary for Rambler and provides context for how each term is used within the architecture.

---

# How to Use This Glossary

Each entry includes:

- **Definition** – What the term means.
- **Architectural Context** – How the term is used within Rambler.
- **Related Concepts** – Other terms closely associated with it.

---

# Candidate

**Definition**

A possible pronunciation-preserving transformation generated by the Rambler Engine.

**Architectural Context**

Candidates are produced during the Generation Layer and evaluated before they are exposed through the Public API.

**Related Concepts**

- Resegmentation
- Scoring
- Validation

---

# Deterministic

**Definition**

A property where identical inputs, language resources, and configuration always produce identical outputs.

**Architectural Context**

Determinism is one of the fundamental architectural principles of Rambler.

**Related Concepts**

- Validation
- Scoring
- Language Pack

---

# Dictionary Model

**Definition**

The knowledge component responsible for providing lexical information to the engine.

**Architectural Context**

A Dictionary Model belongs to a Language Pack and never performs processing.

**Related Concepts**

- Language Pack
- Knowledge Layer

---

# Evaluation Layer

**Definition**

The architectural layer responsible for assessing generated candidates.

**Architectural Context**

It includes the Scoring Engine, Difficulty Engine, and Validation Engine.

**Related Concepts**

- Generation Layer
- Delivery Layer

---

# Generation Layer

**Definition**

The architectural layer responsible for discovering possible pronunciation-preserving transformations.

**Architectural Context**

It performs generation but not evaluation.

**Related Concepts**

- Evaluation Layer
- Pronunciation Architecture

---

# Knowledge Layer

**Definition**

The architectural layer containing language-specific information.

**Architectural Context**

Knowledge Components supply information without performing processing.

**Related Concepts**

- Language Pack
- Dictionary Model

---

# Language Pack

**Definition**

A self-contained collection of language-specific resources required by the Rambler Engine.

**Architectural Context**

Language Packs allow the engine to remain language-independent.

**Related Concepts**

- Dictionary Model
- Pronunciation Resources

---

# Pronunciation Preservation

**Definition**

The architectural objective of maintaining spoken similarity while transforming written form.

**Architectural Context**

This principle distinguishes Rambler from conventional text transformation engines.

**Related Concepts**

- Pronunciation Architecture
- Candidate

---

# Public API

**Definition**

The architectural contract between Rambler and consuming applications.

**Architectural Context**

The Public API exposes engine capabilities rather than internal implementation details.

**Related Concepts**

- Delivery Layer
- Architectural Contract

---

# Rambler Engine

**Definition**

A reusable linguistic engine that generates pronunciation-preserving transformations.

**Architectural Context**

Rambler is designed as a reusable capability within the StudioOS ecosystem rather than as a single-purpose application.

**Related Concepts**

- StudioOS
- Language Pack
- Public API

---

# Resegmentation

**Definition**

The process of reorganizing pronunciation into alternative written structures while preserving spoken form.

**Architectural Context**

Resegmentation is Rambler's primary generation mechanism.

**Related Concepts**

- Candidate
- Pronunciation Preservation

---

# Responsibility Matrix

**Definition**

The architectural reference describing subsystem ownership, inputs, outputs, and boundaries.

**Architectural Context**

It serves as the architectural index for the Rambler Engine.

**Related Concepts**

- Architectural Boundaries
- Processing Pipeline

---

# StudioOS

**Definition**

The modular platform within which Rambler is developed and integrated with other reusable engines.

**Architectural Context**

StudioOS emphasizes reusable capabilities over isolated applications.

**Related Concepts**

- Engine
- Public API

---

# Validation Boundary

**Definition**

The architectural transition between internally evaluated candidates and externally consumable outputs.

**Architectural Context**

Only validated candidates cross this boundary.

**Related Concepts**

- Validation Engine
- Public API

---

# Architectural Vocabulary

Several concepts introduced throughout the documentation apply beyond Rambler and are expected to become part of the broader StudioOS architectural language.

Examples include:

- Stable Core
- Extension Surface
- Internal Reality
- Architectural Contract
- Responsibility First
- Knowledge Component
- Processing Component
- Interface Component
- Generation Layer
- Evaluation Layer
- Delivery Layer

These concepts should remain consistently defined across future StudioOS engines.

---

# Maintaining the Glossary

As Rambler evolves, new architectural concepts should be added to this glossary.

Existing definitions should be refined rather than duplicated.

Maintaining a consistent vocabulary is essential for preserving architectural clarity.

---

# Success Criteria

The glossary succeeds when:

- contributors share a common architectural vocabulary,
- terminology remains consistent across documents,
- new contributors can understand the architecture more quickly,
- architectural discussions use precise and unambiguous language.

---

# Architect's Notes

A mature architecture is supported not only by diagrams and specifications but also by a shared language.

This glossary serves as the vocabulary of the Rambler Engine and provides the conceptual foundation for future contributors and future StudioOS engines.
